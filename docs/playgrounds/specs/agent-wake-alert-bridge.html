Agent Wake Alert Bridge

This playground maps the proposed bridge from MetaTrader 5 alerts to Codex, Claude Code, Antigravity, MT5 push notifications, and optional trade execution. It prevents the main anti-pattern: treating an agent wake as implicit permission to trade.

Current behavior is read-heavy: mt5 alert list reads alerts.dat, and the default MCP server exposes read and dry-run tools only. Target behavior adds a policy engine before any mutation can happen.

Wake Pipeline

1. Observe

MT5 Event Source

Reads terminal alerts, market state, or trusted strategy signal files.

No outbound writes to alerts.dat.

2. Normalize

wake.v1

Builds a stable event id, dedupe key, trigger payload, account context, and source.

3. Decide

Policy Engine

Matches user rules and selects notify, ask, dry-run, or trade behavior.

4. Wake

Adapters

Dispatches to Codex, Claude Code, Antigravity, webhook, and MT5 push sinks.

5. Execute

Trade Executor

Calls existing order and position library functions only after dry-run and policy pass.

Permission Lanes

Notify Only

Send user notifications and write audit records. No trade intent is executed.

Ask Permission

Dry-run a configured trade intent, wake the agent, ask the user, and wait for approval.

Auto Dry-Run

Validate the proposed trade automatically and notify the user of pass/fail results.

Auto Trade

Mutates only after dry-run, policy, dedupe, account, and live gates pass.

Safety Contract

Real-account execution keeps the existing triple gate: cfg["live"] = true, MT5_LIVE=1, and caller live intent. For a daemon, caller live intent must come from an explicit live startup flag or an equivalent local permission-bound adapter field.

The default mt5-mcp server stays read and dry-run only. A mutation-capable agent surface must be separate or explicitly armed by a follow-up spec.

MT5 Mobile Push Path

MT5 can send messages to the mobile app through MetaQuotes ID push notifications. For CLI-originated messages, the preferred bridge is an MQL5 service or EA that reads a local queue file and calls SendNotification() inside the terminal.

This avoids writing the binary alert store and lets the project test a queue contract, message length validation, and rate-limit handling.

Spec Seed

Interfaces

wake.v1 events, trade_intent.v1 actions, policy config, adapter results, and JSONL audit records.

Build Order

Read-only watcher, notification adapters, agent adapters, permissioned executor, then MCP mutation decision.

Acceptance Tests

Policy validation, dedupe, ask blocks mutation, auto dry-run stops, auto trade dry-runs first, real-account gate matrix, adapter failure isolation.

Non-Goals

No strategies, no alert-store mutation, no live gate bypass, no default MCP live trading.

Source Anchors

  • mt5/cli.py: envelope contract, alert command, live-intent threading.
  • mt5_cli/alert/alert.py: read-only alert parser and path resolution.
  • mt5_mcp/server.py: current read and dry-run MCP surface.
  • mt5_cli/orders/orders.py: order mutation functions and live gate.
  • mt5_cli/positions/positions.py: position mutation functions and live gate.
  • Codex Automations: scheduled and same-thread recurring tasks.
  • Claude Code Hooks: lifecycle hooks and permission decisions.
  • Antigravity Hooks: execution-loop hooks.
  • MQL5 SendNotification: MT5 mobile push function.
